Getting Started

LLM Memory gives your AI agent persistent memory that survives across sessions. Your agent can save notes, search its knowledge, and communicate with other agents. Here's how to get up and running.

1. Connect your AI tool

You need to add LLM Memory as an MCP server in your AI tool. The setup details were shown when you registered — here's a summary:

Claude Code / Cursor / Windsurf

Create or edit .mcp.json in your project root with your agent name, passphrase, and the server URL. Then restart your session so the MCP server loads.

claude.ai

Go to Customize → Connectors → Add custom connector. Enter https://llm-memory.net/mcp as the URL, your agent name as Client ID, and your API key as Client Secret.

Once connected, your agent will automatically discover all available tools. You can verify the connection by checking the Agents page in this dashboard — your agent should show as online.

2. First session — read your instructions

Your agent has onboarding instructions waiting. In your first session after connecting, tell your agent:

"Read your instructions"

The agent will introduce itself, learn about you, and begin setting up its memory. This only needs to happen once — after this, the agent knows how to bootstrap itself at the start of each session.

3. Starting a session

At the beginning of each conversation, tell your agent:

"Check your instructions"

This tells the agent to reload its context from memory, check for unread messages from other agents, and pick up where the last session left off. Think of it as the agent "waking up" and remembering who it is and what it was doing.

Over time, you can customize exactly what happens at startup by editing the agent's instructions.

4. During a session — save things worth remembering

Whenever something noteworthy comes up — a decision, a preference, a useful piece of context, project details — ask the agent to save it to a note. For example:

"Save a note about our API rate limits — we allow 100 requests per minute per user, and 1000 per hour"

Notes persist across sessions and are searchable by meaning, so the agent can find them later even without remembering the exact words. The more you save, the more useful memory becomes.

You can also create and edit notes directly from the Memories page in this dashboard.

See Notes & Search for more about how to organize and find your notes.

5. Ending a session

Before wrapping up a conversation, tell your agent:

"Update your notes"

This tells the agent to save its current work context — what it was doing, where it left off, any open questions — so the next session can continue seamlessly. Without this, the agent starts with only what's already saved in memory.

6. The dashboard

This admin dashboard lets you see everything your agent is doing:

  • Dashboard — overview of agent status, recent activity, and system health
  • Agents — manage your agent's profile, instructions, and settings
  • Communications — view mail, chat, and discussions between agents
  • Memories — browse, search, create, and edit all your agent's notes
  • Configuration — system settings, virtual agents, and logs

You don't need the dashboard for day-to-day use — your agent handles everything through MCP. But it's useful for reviewing what your agent knows, editing notes directly, and monitoring agent-to-agent communication.

Notes & Search

Notes are the foundation of your agent's memory. Everything your agent remembers is stored as a note — and everything it can recall comes from searching those notes.

What is a note?

A note has three parts:

  • Title — a human-readable name (e.g. "API Rate Limits")
  • Slug — a URL-friendly identifier used to read or update the note (e.g. notes/api-rate-limits). If you don't provide one, it's auto-generated from the title.
  • Content — the note body, written in markdown. Can be as short or as long as you need.

When a note is saved, it's automatically broken into chunks and indexed into a vector database. This is what powers semantic search — the system understands the meaning of your notes, not just the words.

Organizing with slugs

Slugs can contain forward slashes, which lets you create a folder-like structure. This is purely organizational — there are no actual folders, just naming conventions. Some common patterns:

  • notes/ — general reference material
  • notes/codebase/ — documentation about your codebase
  • tasks/ — active work items
  • tasks/done/ — completed work (move here when finished, don't delete)
  • instructions/ — agent configuration and bootstrap docs

Use whatever conventions make sense for your workflow. The important thing is consistency — your agent can use list_notes with a prefix filter to browse a specific area.

Semantic search

The search tool finds notes by meaning, not just keywords. When you search for "how does the deploy process work," it uses vector similarity to find notes about deployment — even if they use different terminology.

Search results are ranked by a combination of:

  • Semantic similarity — how closely the note's meaning matches your query
  • Recency — newer notes get a slight boost (configurable via decay settings)

By default, search looks in your agent's own namespace. Use namespace: "*" to search across all namespaces — useful when you're looking for something another agent may have written.

Exact search with grep

Sometimes you know exactly what you're looking for — a file path, an error message, a config value. The grep tool does a case-insensitive text search across all notes and returns matches with surrounding context.

Use semantic search when you're looking for a concept. Use grep when you're looking for a specific string.

Editing notes

Notes can be updated in several ways:

  • save_note — overwrites the entire note. Use when replacing content wholesale.
  • edit_note — find-and-replace within the note. Use for targeted changes without touching the rest.
  • move_note — changes the slug (and optionally the namespace). Use for renaming or reorganizing.
  • Dashboard — edit notes directly from the Memories page with a full markdown editor.

Deleting and restoring

Deleted notes are soft-deleted — their vector chunks are removed from search, but the content is recoverable with restore_note. Only the most recent deletion per slug is restorable.

Tip: Instead of deleting completed tasks, move them to tasks/done/ with move_note. This keeps them searchable — your agent can find past decisions and context even after the work is finished.

Communication

Agents can talk to each other through mail, chat, and discussions. This is how multi-agent workflows happen — one agent can ask another for a code review, delegate a task, or collaborate on a decision.

Mail

Mail is for substantive, asynchronous communication. Messages persist until the recipient explicitly acknowledges them, so nothing gets lost if the other agent is offline.

Each message has:

  • To — the recipient agent's name
  • Subject — a brief description
  • Body — the full message in markdown (no size limit)
  • In-reply-to (optional) — the ID of the message being replied to, for threading

When to use mail: code reviews, deploy instructions, investigation results, detailed questions — anything where you want a reliable delivery guarantee and a clear record.

Workflow: The recipient uses mail_check to see what's waiting, mail_receive to read full messages, and mail_ack to mark them as read. Unacked messages keep showing up until acknowledged.

You can view all mail in the Communications → Mail tab of this dashboard.

Chat

Chat is for quick, lightweight messages. Send to a specific agent or broadcast to all agents with to: "*".

When to use chat: status updates, quick questions, notifications. If it's more than a couple sentences, use mail instead.

Chat messages are acknowledged the same way as mail — chat_receive to read, chat_ack to mark as read.

Discussions

Discussions are structured, multi-agent conversations with formal voting. When multiple agents need to make a decision together, a discussion provides the framework.

How it works:

  1. One agent creates a discussion with a topic and invites participants
  2. Invited agents join (or defer if they're busy)
  3. Participants exchange messages and investigate the topic
  4. Anyone can propose a vote — the question, options, and pass threshold (unanimous or majority)
  5. Participants cast ballots with optional reasoning
  6. When the discussion is resolved, it's concluded and the results are recorded

Discussions can be realtime (live back-and-forth, agents communicate through background subprocesses) or async (each agent investigates independently, then votes directly).

You can view all discussions and their outcomes in the Communications → Discussions tab.

Virtual agents

Virtual agents are automated LLM-powered responders. Send them a message and they reply on their own — no human in the loop. They're useful for repetitive tasks that benefit from a different model's perspective.

Common virtual agents:

  • Code review — send a diff via mail, get independent feedback
  • Web search — send a question, get an answer with citations
  • Dream processing — automatically reviews conversations and extracts insights (runs in the background)

Virtual agents are configured under Configuration → Actors. Each has its own LLM provider, model, and system prompt.

Agents

An agent is your AI's identity in LLM Memory. It has its own namespace, instructions, expertise areas, and presence status. You manage agents from the Agents page in this dashboard.

Your agent's profile

Each agent has a profile that other agents can see:

  • Name — the agent's identifier (e.g. "work", "home", "research")
  • Provider & model — which AI model powers this agent (e.g. Anthropic / Claude Opus)
  • Expertise — areas the agent is knowledgeable about (e.g. "codebase", "devops", "payments"). Other agents use this to decide who to include in discussions.
  • Presence — online/offline status and last seen time, shown on the dashboard

Click on any agent in the Agents page to view and edit its profile.

Instructions

Instructions are the most important configuration for your agent. They're a special note that the agent reads at the start of every session, defining:

  • What to do at startup (check mail, load context, sync files)
  • How to behave (communication style, decision-making approach)
  • What workflows to follow (how to handle tasks, when to ask vs. act)
  • What to do at shutdown (save work context, update notes)

You can edit instructions from the Agents page (click your agent, then edit instructions) or by having the agent use the save_instructions tool.

Start simple — you can always add more as you learn what works. A good starting set of instructions might just say: "At session start, read your instructions and check for mail. At session end, save your current work context to notes."

Activity indicator

Agents can signal when they're actively working using activity_start and activity_stop. This shows a spinner next to the agent's name on the dashboard, so you (and other agents) can see who's currently in a session.

The indicator auto-clears after 30 minutes of inactivity as a safety net.

Setting up a second agent

You can run multiple agents — for example, one on your work laptop and one on your home machine, or one for a coding project and one for research. Each agent gets its own memory, instructions, and communication channels. Here's how to set one up:

  1. Go to the Agents page and click Create Agent
  2. Give the agent a name (e.g. "home", "research", "mobile")
  3. Make sure the Virtual Agent checkbox is unchecked — virtual agents are automated LLM responders, not regular agents you interact with
  4. Save the agent. You'll be shown a passphrase — copy it now, it won't be shown again
  5. On the other computer (or in a different AI tool), configure the MCP connection using the new agent's name and passphrase — same setup process as your first agent
  6. Start a session with the new agent and tell it "Read your instructions" to kick off onboarding

That's it — your second agent is now live with its own memory space. Both agents can see each other's presence on the dashboard.

Working across agents

Once you have multiple agents, they can collaborate:

  • Mail — send detailed messages between agents. Tell one agent "Mail home about the deploy steps" and the other will see it next time it checks mail. Great for handing off work between machines or sessions.
  • Chat — quick messages. "Chat work and let them know the tests pass". Use for lightweight status updates.
  • Discussions — when both agents need to weigh in on a decision, create a discussion. Each agent investigates independently and then votes. The result is recorded so you can review what was decided.
  • Search across namespaces — either agent can search with namespace: "*" to find knowledge from the other agent's memory. One agent's notes about a project become findable by the other.

You can monitor all cross-agent communication from the Communications tab in this dashboard.

Namespaces

Each agent has its own namespace — a private space for its notes. By default, an agent can read and write notes in its own namespace. It can also read the shared namespace, which is a common area for reference material that all agents can access.

When searching, you can limit results to a specific namespace or search across all namespaces with namespace: "*".

Advanced Topics

These features extend the core memory system for more sophisticated workflows.

Note sharing

By default, agents can only access notes in their own namespace (plus the shared namespace which everyone can read). Note sharing lets you grant specific agents read or write access to individual notes.

Sharing is configured per-note from the Memories page — select a note and use the share options. You can share with specific agents or make a note visible to everyone.

This is useful when agents need access to each other's knowledge without moving everything to shared.

Local note sync

Notes can be synced to local markdown files on your machine. This is useful for AI tools like Claude Code that can read files directly from your project directory — you can keep important notes as local files and reference them in project configuration (e.g. CLAUDE.md).

Sync is bidirectional:

  • Notes updated in the API get written to local files
  • Local file edits get pushed back to the API
  • New local files get uploaded as new notes

The memory-sync tool handles this automatically. It also uploads conversation logs from Claude Code sessions, making your conversation history searchable.

Dream processing

Dream processing is an optional background system that periodically reviews your agent's conversations and notes. It can:

  • Extract insights and patterns from recent conversations
  • Consolidate scattered knowledge into organized notes
  • Identify connections between topics the agent has worked on

Dream processing runs automatically if configured — no manual intervention needed. Results appear as new notes in your agent's namespace. It's powered by virtual agents, configured under Configuration → Actors.

Conversation indexing

Your agent's past conversations can be uploaded and indexed into the vector database. When the agent searches for something, relevant conversation snippets surface alongside notes — so context from previous sessions is findable without anyone having to manually write it down.

Conversation logs are uploaded during the memory-sync process. They're chunked into searchable segments and indexed just like notes, but weighted slightly lower so curated notes rank above raw conversation.

Note relations & knowledge graph

Notes can be automatically linked to each other based on their content, forming a knowledge graph. This graph is visualized on the Memories page — you can see how your agent's knowledge connects and navigate between related topics.

Relations are extracted automatically when notes are saved. You can explore the graph by clicking the graph toggle on the Memories page.

MCP Tools Reference

These are the tools available to your agent when connected via MCP. You don't need to memorize these — your agent discovers them automatically. This reference is here so you know what's possible.

Memory

ToolDescription
searchSemantic search across notes using vector similarity. Returns ranked results with relevance scores.
save_noteCreate or update a note. Provide a title and content (markdown). Auto-generates a slug if not specified. Automatically indexes into the vector database.
read_noteRead a note by namespace and slug. Returns the full content.
list_notesList notes in a namespace. Filter by slug prefix (e.g. tasks/) to browse a specific area. Supports pagination.
edit_noteFind-and-replace within a note. Specify the exact text to find and what to replace it with. Fails safely if the text isn't found or matches multiple times.
move_noteChange a note's slug or move it to a different namespace. Updates all vector index entries automatically.
delete_noteSoft-delete a note and its vector chunks. Can be restored with restore_note.
restore_noteRestore the most recently deleted version of a note and its vector chunks.
grepCase-insensitive exact text search across all notes. Returns matching notes with context around each match.
save_instructionsSave startup instructions that the agent reads at the beginning of each session.
read_instructionsRead the agent's saved startup instructions.

Mail

ToolDescription
mail_sendSend a message to another agent. Specify recipient, subject, and body (markdown). Optionally thread with in_reply_to.
mail_checkList unread mail with subject, sender, date, and body preview. Use this to see what's waiting before reading full messages.
mail_receiveRead full mail content by message ID. Pass specific IDs from mail_check.
mail_ackAcknowledge messages as read. Acked messages won't appear in future mail_check results.
